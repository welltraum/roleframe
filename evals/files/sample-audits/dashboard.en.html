supervisor

Mixes routing and final answer without typed contracts.

12/30Working prototype
Full auditsupervisor.mdsupervisor.txtfactory.py
Short verdict

The supervisor carries both routing and final response assembly. That blurs the function boundary, and the contracts do not read like an engineering interface. The main risk comes from the combination of free-form output, hidden dependency, and implicit handoff.

IDEF0
Input: The agent accepts a user request in the tax domain and distinguishes cases with documents from those without them. The input type is not described as a schema.
Control: The prompt defines a sequence of classification, delegation, and final answer assembly. Constraints for honest degradation and output formatting are not formalized.
Mechanism: The frontmatter declares the `docs-search` toolkit, two sub-agents, and retry middleware. The real toolset comes from the factory and does not match the prompt text.
Output: The output is described as useful free text. There is no machine-readable contract for success, empty, error, or delegation.
Key evidence points
  • evals/files/sample-agents/prompts/agents/supervisor.txt#L3-L5, The routing contract exists only as a prose rule without a typed payload.
  • evals/files/sample-agents/prompts/agents/supervisor.txt#L6, The prompt allows `archive_search`, which is missing from the declared toolkit.
  • evals/files/sample-agents/src/tools/factory.py#L1-L4, The factory confirms that `docs-search` contains only `search_documents` and `fetch_document`.
  • evals/files/sample-agents/prompts/agents/supervisor.txt#L8-L9, The final answer is allowed in free text and may fill in gaps.

document-finder

No typed handoff and no formal failure policy.

18/30Working prototype
Full auditdocument-finder.mdfactory.py
Short verdict

Document-finder is already close to an atomic business function. The main gaps are not in the agent's purpose, but in the handoff and failure policy. For now the result is good enough for a prototype, but not for a reliable multi-agent contract.

IDEF0
Input: The agent receives a document search request from the supervisor. The input format is expected in text rather than a schema.
Control: The prompt requires finding relevant documents and reporting an empty result. Retry and handoff rules are described heuristically.
Mechanism: The `docs-search` toolkit provides search and fetch capabilities. The runtime policy for timeout, retry count, and budget is not fixed.
Output: The agent returns the best documents or a message that nothing was found. There is no machine-readable payload with status and a bounded result set.
Key evidence points
  • evals/files/sample-agents/agents/document-finder.md#L11-L14, The happy path is described, but the handoff is explicitly allowed without a fixed schema.
  • evals/files/sample-agents/agents/document-finder.md#L12, Retry is defined as “if it feels worth it” rather than as a verifiable policy.
  • evals/files/sample-agents/src/tools/factory.py#L1-L4, The toolkit confirms that the agent has enough tools for the search function.
  • evals/files/sample-agents/agents/document-finder.md#L13-L14, The empty-result and success paths exist only as a prose response back to the supervisor.

response-drafter

The prompt encourages guesswork and does not provide an honest partial path.

16/30Working prototype
Full auditresponse-drafter.mdfactory.py
Short verdict

Response-drafter is isolated as a separate function and is not mixed with document search. The main problem is concentrated in Control: the prompt pushes the model to produce a complete answer even on incomplete data. In a tax scenario, that makes the agent unsafe by design.

IDEF0
Input: The agent accepts documents and confirmed facts from other agents. The input format is implied, but not described as a schema.
Control: The prompt requires a full answer and allows filling gaps from context. Limits on sources of truth and an honest refusal path are missing.
Mechanism: The `drafting` toolkit is minimal and matches the response-formatting task. The risk comes not from the tools, but from the prompt policy.
Output: The output is flexible natural language without a strict schema. There is no defined status for partial, empty, or error paths.
Key evidence points
  • evals/files/sample-agents/agents/response-drafter.md#L11, The prompt allows filling gaps from context rather than only from confirmed facts.
  • evals/files/sample-agents/agents/response-drafter.md#L12, A strict result schema is not required, which makes downstream parsing fragile.
  • evals/files/sample-agents/src/tools/factory.py#L3, The toolkit itself does not create the risk, so the problem is localized in Control.
  • evals/expected-findings.md#L13-L18, The baseline review explicitly expects a finding about unsafe completion behavior.

4. Evaluation and observability exist only at the fixture-review level, not as per-agent contracts.

Roadmap

Phase 1. Function + contract

Split routing and final assembly, then introduce typed contracts at every cross-agent boundary.

Phase 2. Runtime + safety

Add bounded retry, timeout, refusal, and partial paths where only prose heuristics exist today.

Phase 3. Eval + observability

Close agent-level regressions, route telemetry, and signals for unsafe completion.
