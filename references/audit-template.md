# Шаблон архитектурного аудита агента

## Быстрая карта разделов

- Принцип аудита
- Правила оформления
- Формат доказательства
- Карта артефактов
- Граница ответственности
- Минимальная бизнес-функция
- IDEF0 и Input/Output
- Control
- Mechanism / runtime
- Traceability, eval, operations, change management
- Конфликты, scoring, backlog, patch plan, verdict

Используй для генерации файла аудита. Путь: `docs/agent_audit/NN_name.md`.

## Принцип аудита

**Аудит собирается по коду и артефактам, а не по документации.**
Единственный источник истины — исходный код: промпты (`.md` с frontmatter), runtime (`.py`), конфигурации tool factory, тесты, eval-сценарии. Если документация утверждает одно, а код делает другое — фиксируй расхождение, верь коду.

## Правила оформления

1. Весь файл — на языке запроса.
2. Избегай английских вставок там, где есть хороший русский эквивалент, кроме устоявшихся терминов и названий протоколов/компонентов (happy path, runtime, middleware, SOP, eval, MCP).
3. Каждый вывод должен быть проверяемым: он обязан опираться на **конкретный артефакт** и сопровождаться ссылкой вида `файл:строка` или диапазоном `файл:L10-L24`.
4. Если вывод основан на **отсутствии** нужного артефакта, это тоже фиксируется как доказательство: `не найдено в [список файлов]`.
5. Любое замечание должно быть локализовано по слою: **Input / Control / Mechanism / Output / Evaluation / Operations / Change management**.
6. Аудит должен отвечать на три вопроса:
   - Из каких артефактов состоит агент.
   - Где именно локализован дефект.
   - Что именно нужно изменить и как проверить изменение.
7. **Проверяй реальный toolkit, а не текст промпта.** Промпт может ссылаться на несуществующий tool. Чтобы узнать, какие tools реально доступны агенту, смотри:
   - YAML frontmatter → `tools.toolkits` — список toolkit-ов
   - Tool factory (`src/tools/factory.py` или аналог) — что включает каждый toolkit
   - Если промпт вызывает tool, которого нет в toolkit — это AP-14 (Hidden Dependency)

## Формат доказательства

Каждый существенный вывод оформляй через один из трёх типов доказательства.

### 1. Прямая цитата

```
Артефакт: [Control / Mechanism / Evaluation / Operations / Change management]
Источник: `path/to/file#L10-L18`
Тип фрагмента: [prompt / skill / tool / runtime / schema / test / log / config]
Вывод: [что именно доказывает фрагмент]

> дословная цитата 2-8 строк
```

### 2. Структурный индикатор

```
Артефакт: [слой]
Источник: `path/to/file#L...`
Тип фрагмента: [код / схема / конфиг]
Вывод: [какой вывод следует из структуры реализации]
Комментарий: [почему это доказывает замечание]
```

### 3. Отсутствие артефакта

```
Артефакт: [слой]
Проверено: [список файлов / директорий / разделов]
Не найдено: [что именно отсутствует]
Вывод: [какой инженерный риск из этого следует]
```

## 5. Слой Mechanism / runtime

**Важно:** Состав инструментов определяй по двум независимым источникам:
1. **YAML frontmatter** агента → `tools.toolkits` — какие toolkit-ы подключены
2. **Tool factory** (например `src/tools/factory.py`) → что конкретно входит в каждый toolkit

Если промпт упоминает tool, которого нет в toolkit — это расхождение Control↔Mechanism.
Если toolkit содержит tool, который промпт не упоминает — это скрытая возможность.

## 11. Оценка зрелости (10 критериев)

Шкала:
- **0** — отсутствует (не найдено ни в prompt, ни в коде)
- **1** — есть частично (упомянуто в prompt, но не поддержано кодом; или наоборот — есть в коде, но не описано)
- **2** — описано и реализовано (prompt + код согласованы), но нет автоматической проверки
- **3** — описано, реализовано и проверяется (есть eval-сценарий или тест)

**Правило:** Колонка «Обоснование» обязана содержать `файл:строка` для баллов ≥1 и `не найдено в [файлы]` для балла 0. Без привязки к коду балл недействителен.

## 15. Бэклог изменений

Порядок приоритета: функция → control → mechanism → evaluation → operations.

| # | Действие | Слой | Артефакт | Файл:строка | Приоритет | Риск регрессии |
|---|---|---|---|---|---|---|
| 1 | … | Control | системный промпт | `...` | P1 | Safe / Breaking / Behavioural |
| 2 | … | Mechanism | runtime | `...` | P1 | … |
| 3 | … | Evaluation | dataset | `...` | P2 | … |

**Определения риска регрессии:**
- **Safe** — аддитивное изменение, не ломает существующий интерфейс
- **Breaking** — меняет контракт, формат выхода или внешний интерфейс
- **Behavioural** — меняет логику решений, маршрутизацию, SOP или поведение агента

## 17. Вердикт

[2–4 предложения в активном залоге:]
- какой это уровень зрелости;
- где главный системный разрыв;
- какой первый шаг даст наибольший эффект;
- почему именно этот шаг нужно делать раньше остальных.

Пример формата:

> Агент ближе к рабочему прототипу, чем к управляемому компоненту. Главный разрыв находится в output contract: системный промпт не задаёт машинный формат ответа, а runtime не валидирует результат. Из-за этого агентом нельзя безопасно пользоваться как частью более широкой системы. Первое изменение — ввести строгую схему ответа и post-validation, а затем закрыть её негативными eval-сценариями.
