# Методология разработки агента

Документ основан на материалах `Инженерия агентов 2026`, прежде всего на разделах про инструкции, требования к агенту и IDEF0. Он нужен как короткая, но инженерно внятная опора для чтения dashboard и audit-файлов.

## 1. Агент - это не LLM

LLM сама по себе не агент. Это стохастическое вычислительное ядро, которое интерпретирует текст и предсказывает следующий токен. Агент начинается там, где вокруг LLM появляется исполняющая среда:

- память;
- инструменты;
- ограничения;
- orchestration;
- output contract;
- eval и observability.

Если этого нет, перед нами не агентная система, а обычный request-response интерфейс к модели.

## 2. Два контура разработки

В разработке агента есть два связанных, но разных контура.

### Инженерный контур

Он отвечает за:

- архитектуру и границы компонентов;
- интеграции с tools и внешними API;
- runtime loop;
- масштабирование и отказоустойчивость;
- observability;
- воспроизводимость и эксплуатацию.

Без него прототип остаётся "магией", которую нельзя безопасно развивать в проде.

### Исследовательский контур

Он отвечает за:

- роль агента;
- стиль мышления;
- SOP;
- constraints;
- форматы input/output;
- eval dataset и анализ ошибок.

Без него агент может быть формально корректным, но по факту бесполезным.

Хорошая разработка держит оба контура одновременно. Если оставить только инженерный, получится пустая автоматика. Если оставить только исследовательский, получится хрупкий промпт-прототип.

## 3. Минимальная единица проектирования

Практическое правило:

> Агент = одна бизнес-функция.

Не "умный помощник", не "агент, который делает всё", а одна функция с ясным началом, ограничениями и конечным продуктом.

Причины простые:

1. Функцию можно протестировать.
2. Для функции можно задать ясный output contract.
3. Для функции можно понять, где заканчивается ответственность.

Если задачу нельзя описать одной атомарной функцией, её лучше декомпозировать на несколько агентов или на supervisor + sub-agents.

## 4. IDEF0 как каркас требований

IDEF0 удобно использовать как каркас для требований к агенту.

### Input

Что запускает функцию:

- пользовательский запрос;
- событие;
- файл;
- handoff от другого агента.

Здесь важно не просто перечислить input, но и выделить классы запросов, edge cases и недопустимые состояния.

### Control

Что управляет поведением агента:

- role / identity;
- SOP;
- constraints;
- output contract;
- decision rules.

Это главный управляющий слой. Именно он превращает вероятностное поведение модели в узкий набор допустимых действий.

### Mechanism

Чем агент исполняет функцию:

- tools;
- runtime;
- memory;
- sandbox;
- sub-agents;
- интеграции.

Mechanism не должен смешиваться с control. Prompt не должен тащить в себя низкоуровневые детали API, retry counts и другие технические подробности, которым место в коде и конфигурации.

### Output

Что считается завершённым результатом:

- успешный результат;
- бизнес-пустота;
- техническая ошибка;
- partial / refusal;
- delegation.

Выход должен быть формализован как интерфейс, а не как "какой-то полезный текст".

## 5. Что должно быть в control-слое

### Identity & Goal

Агент должен понимать:

- кто он;
- какую одну функцию выполняет;
- для кого производит результат.

Single Responsibility Principle здесь обязателен. Слишком широкая роль почти всегда приводит к дрейфу поведения.

### SOP

SOP - это procedural core агента. Он задаёт:

- входное условие;
- последовательность шагов;
- ветвления;
- stop conditions;
- правила использования tools.

Хороший SOP приближает поведение агента к псевдокоду. Плохой SOP оставляет модели слишком много пространства для импровизации там, где нужна точность.

### Constraints

Это список того, чего агент делать не должен:

- не выдумывать данные;
- не интерпретировать техническую ошибку как бизнес-результат;
- не выходить из роли;
- не использовать tool вне разрешённого сценария.

Constraints в первую очередь синхронизируют поведение модели с бизнес-логикой.

### Output contract

Агент - это функция. Значит, выход должен быть похож на API-контракт:

- machine-readable;
- валидируемый;
- стабильный между запусками;
- охватывающий не только happy path.

Если output не типизирован, агент хуже встраивается в систему и его сложнее тестировать.

## 6. Контекст - это инженерная задача

Context engineering нужен не для того, чтобы "засунуть всё в prompt".

### Static context

Это неизменяемая часть:

- системные правила;
- политики;
- tone of voice;
- описания tools;
- доменная база.

### Dynamic context

Это переменная часть:

- текущий запрос;
- история диалога;
- результаты tool calls;
- runtime state;
- дата и время.

### Четыре приёма управления контекстом

- **Write**: выносить длинные промежуточные артефакты во внешнее состояние.
- **Select**: подмешивать только релевантные куски контекста.
- **Compress**: сокращать сырые tool outputs до полезного сигнала.
- **Isolate**: разделять ответственность через sub-agents и sandbox.

## 7. Гигиена разработки

Промпт - это артефакт разработки.

Это означает:

- хранить его в репозитории;
- проводить code review;
- версионировать;
- собирать модульно;
- прогонять регрессии после правок.

Иногда одна строка в prompt меняет поведение системы не слабее, чем одна строка в коде.

## 8. Что проверять на eval

Для агента мало пары happy-path примеров.

Нужно проверять:

- позитивные сценарии;
- edge cases;
- business-empty;
- tool failure;
- partial / refusal;
- incorrect routing;
- нарушение constraints.

Eval dataset должен содержать не только "приятные" кейсы, но и те, которые реально ломают логику.

## 9. Практический минимум требований к агенту

Перед реализацией у агента должны быть хотя бы такие артефакты:

1. Одна чёткая бизнес-функция.
2. IDEF0-описание: input / control / mechanism / output.
3. Role + SOP + constraints.
4. Typed output contract.
5. Список tools и правила их использования.
6. Понимание edge cases.
7. Eval-сценарии.
8. Базовые операционные сигналы.

Если этих артефактов нет, команда почти наверняка перейдёт в режим "тюним prompt на ощущениях".
