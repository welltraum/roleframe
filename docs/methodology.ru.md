# Методология разработки агента

Этот документ собран по материалам `Инженерия агентов 2026`, в первую очередь по разделам про инструкции, требования к агенту и IDEF0. Его задача, дать короткую, но инженерно полную опору для чтения dashboard и audit-файлов.

## 1. Агент, это не LLM

LLM сама по себе не является агентом. Это стохастическое вычислительное ядро, которое умеет интерпретировать текст и генерировать следующий токен. Агент начинается там, где вокруг LLM появляется исполняющая среда:

- память;
- инструменты;
- ограничения;
- orchestration;
- output contract;
- eval и observability.

Если этого нет, перед нами не агентная система, а request-response интерфейс к модели.

## 2. Два контура разработки

Разработка агента делится на два связанных, но разных контура.

### Инженерный контур

Он отвечает за:

- архитектуру и границы компонентов;
- интеграции с tools и внешними API;
- runtime loop;
- масштабирование и отказоустойчивость;
- observability;
- воспроизводимость и эксплуатацию.

Без него прототип остаётся “магией”, которую нельзя безопасно развивать.

### Исследовательский контур

Он отвечает за:

- роль агента;
- стиль мышления;
- SOP;
- constraints;
- форматы input/output;
- eval dataset и анализ ошибок.

Без него агент будет формально корректным, но бесполезным.

Нормальная разработка держит оба контура одновременно. Если оставить только инженерный, получится пустая автоматика. Если оставить только исследовательский, получится хрупкий промпт-прототип.

## 3. Минимальная единица проектирования

Практический принцип:

> Агент = одна бизнес-функция.

Не “умный помощник”, не “агент, который делает всё”, а одна функция с ясным началом, ограничениями и конечным продуктом.

Это нужно по трём причинам:

1. Функцию можно протестировать.
2. Для функции можно задать ясный output contract.
3. Для функции можно понять, где заканчивается ответственность.

Если задачу нельзя описать одной атомарной функцией, это сигнал к декомпозиции на несколько агентов или на supervisor + sub-agents.

## 4. IDEF0 как каркас требований

Требования к агенту удобно собирать через IDEF0.

### Input

Что запускает функцию:

- пользовательский запрос;
- событие;
- файл;
- handoff от другого агента.

Здесь важно не только перечислить input, но и выделить классы запросов, edge cases и недопустимые состояния.

### Control

Что управляет поведением агента:

- role / identity;
- SOP;
- constraints;
- output contract;
- decision rules.

Это главный управляющий слой. Именно он превращает вероятностное поведение модели в узкий коридор допустимых действий.

### Mechanism

Чем агент исполняет функцию:

- tools;
- runtime;
- memory;
- sandbox;
- sub-agents;
- интеграции.

Mechanism не должен смешиваться с control. Prompt не должен описывать низкоуровневые детали API, retry counts и технические подробности, которые лучше держать в коде и конфигурации.

### Output

Что считается завершённым результатом:

- успешный результат;
- бизнес-пустота;
- техническая ошибка;
- partial / refusal;
- delegation.

Выход должен быть формализован как интерфейс, а не как “полезный текст”.

## 5. Что должно быть в control-слое

### Identity & Goal

Агент должен понимать:

- кто он;
- какую одну функцию выполняет;
- для кого производит результат.

Single Responsibility Principle здесь обязателен. Широкая роль почти всегда рождает дрейф поведения.

### SOP

SOP, это procedural core агента. Он задаёт:

- входное условие;
- последовательность шагов;
- ветвления;
- stop conditions;
- правила использования tools.

Хороший SOP приближает поведение агента к псевдокоду. Плохой SOP оставляет модели пространство для импровизации там, где нужна точность.

### Constraints

Это список того, что агент не должен делать:

- не выдумывать данные;
- не интерпретировать техническую ошибку как бизнес-результат;
- не выходить из роли;
- не использовать tool вне разрешённого сценария.

Constraints в первую очередь синхронизируют поведение модели с бизнес-логикой.

### Output contract

Агент, это функция. Значит выход должен быть похож на API-контракт:

- machine-readable;
- валидируемый;
- стабильный между запусками;
- охватывающий не только happy path.

Если output не типизирован, агент плохо встраивается в систему и становится трудно тестируемым.

## 6. Контекст, это инженерная задача

Context engineering нужен, чтобы не пытаться “засунуть всё в prompt”.

### Static context

Это неизменяемая часть:

- системные правила;
- политики;
- tone of voice;
- описания tools;
- доменная база.

### Dynamic context

Это переменная часть:

- текущий запрос;
- история диалога;
- результаты tool calls;
- runtime state;
- дата и время.

### Четыре приёма управления контекстом

- **Write**: выносить длинные промежуточные артефакты во внешнее состояние.
- **Select**: подмешивать только релевантные куски контекста.
- **Compress**: сокращать сырые tool outputs до полезного сигнала.
- **Isolate**: разделять ответственность через sub-agents и sandbox.

## 7. Гигиена разработки

Промпт, это артефакт разработки.

Это означает:

- хранить его в репозитории;
- проводить code review;
- версионировать;
- собирать модульно;
- прогонять регрессии после правок.

Изменение одной строки в prompt может менять поведение системы так же сильно, как изменение кода.

## 8. Что проверять на eval

Для агента недостаточно пары happy-path примеров.

Нужно проверять:

- позитивные сценарии;
- edge cases;
- business-empty;
- tool failure;
- partial / refusal;
- incorrect routing;
- нарушение constraints.

Eval dataset должен содержать не только “приятные” кейсы, но и те, которые реально ломают логику.

## 9. Практический минимум требований к агенту

Перед реализацией у агента должны быть хотя бы такие артефакты:

1. Одна чёткая бизнес-функция.
2. IDEF0-описание: input / control / mechanism / output.
3. Role + SOP + constraints.
4. Typed output contract.
5. Список tools и правила их использования.
6. Понимание edge cases.
7. Eval-сценарии.
8. Базовые операционные сигналы.

Если этих артефактов нет, команда почти неизбежно перейдёт в режим “тюним prompt на ощущениях”.
